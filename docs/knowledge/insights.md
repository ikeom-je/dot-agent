# insights — 運用知見の記録(追記式)

書き方は [README.md](README.md)。見出しはテーマ、新しい知見は末尾に追記。

## モデルの自己申告は過大評価バイアスがある

- 事実: roster-review 初回実行で、Gemini は5領域中4領域を「強」と自己申告した
  (実装実績ゼロの領域を含む)。他薦と実績で補正した結果、採用は実績と一致した領域のみ
  (記録: 2026-07-19)。
- 知見: 担当割りの判断材料は「実績 > 公開情報 > 他薦の一致 > 自己申告」の順で重み付けする。
  自薦より他薦の一致の方が信頼できる。
- 反映: roster-review スキルに反映済み。

## Antigravity(agy)は空応答で失敗することがある

- 事実: 検索委譲で flash×2・pro×1 の計3回連続空応答が発生。別日には1回目空応答→
  2回目成功のケースも。プロンプトの長短との相関は不明(記録: 2026-07-08/20)。
- 知見: 委譲失敗は起こる前提で、上限(`max_delegation_retries`)と代替経路を先に決めておくと
  作業が止まらない。1回の失敗で委譲先を見限らない(翌日は動いた)。
- 反映: cli-routing のフォールバック規則+編成表の代替列に反映済み。

## Codex レビューは的中率が高いが、失敗シナリオの前提が誤っていることがある

- 事実: check-links 実装レビューで「実在ファイルが誤検知される」という major 指摘を
  再現テストしたところ、指摘のメカニズムは誤りだった。ただし同じ指摘に含まれていた
  spec 不整合は正当だった(記録: 2026-07-09)。
- 知見: 失敗シナリオ付きの指摘は反映前に必ず再現を試みる。前提が誤りでも、周辺の指摘
  (仕様不整合など)は正当なことがあるので、指摘を丸ごと棄却しない。
- 反映: cross-review スキル(指摘の再現検証)に反映済み。

## クロスレビューは「作った本人が気づけない欠陥」を実際に拾う

- 事実: 全 bolt でクロスレビュー(Codex中心)を実施し、ほぼ毎回 major を検出。代表例:
  「単一定義箇所を作った bolt で、当の表の書き換え漏れ」(critical)、「圧縮 bolt での
  規定4件の欠落」、「レビュー対象の untracked ファイルが git diff に出ない」
  (記録: 2026-07-06〜20)。
- 知見: 自分の変更意図に沿った見落とし(意図した変更の適用漏れ・やり過ぎ)は自己チェックでは
  捕まらない。2往復・採否記録のプロトコルはコストに見合う。
- 反映: review.md / cross-review スキルの体制として反映済み。

## 圧縮・整理の bolt は「削りすぎ」を必ず疑う

- 事実: コンテキスト圧縮 bolt で、レビューが「条件句の消失による規定の無条件化」など
  4件の欠落を検出した。圧縮後の読解チェック(3モデルに規定5問)は全問正答だったが、
  読解チェックだけでは欠落を検出できなかった(記録: 2026-07-20)。
- 知見: 圧縮の検証は「読める(読解チェック)」と「欠けていない(diff の削除行の精査)」の
  両方が必要。削除行レビューを圧縮 bolt の必須観点にする。
- 反映: 未反映(次に圧縮 bolt をやるときの観点として本ファイルが正)。

## エージェントへの委譲はファイルベースが最も可搬

- 事実: MCP・プラグイン・subagent と経路は環境ごとに違うが、「書面(spec/plan/定義)の
  パス+完了条件を渡し、ダイジェストで受ける」形はどの経路でも同じに保てた。
  ヘッドレスCLIでも同じ型で委譲できる(記録: 2026-07-20)。
- 知見: 委譲の設計はまず「何をファイルとして渡すか」を決め、経路(ツール)は後から差し替える。
- 反映: cli-routing(委譲の必須3点・汎用経路)に反映済み。

## 「単一定義+名前参照」は増築に強い

- 事実: 回数・担当・値を表1箇所に置き、他 doc は名前参照にした結果、値の変更・行の追加が
  1ファイルで完結し続けた。一方、初期にハードコードが数カ所残り、レビューで
  「表と本文の矛盾」として繰り返し検出された(記録: 2026-07-06〜20)。
- 知見: 「数値・担当を本文に書きたくなったら名前参照に置き換える」を書く時点で徹底する方が、
  後からの掃除より安い。
- 反映: AGENTS.md の表運用として反映済み。

## 人間の判断が要る箇所は「ゲート」として明文化すると迷いが消える

- 事実: リリース GO・外部スキル導入・編成変更の3つを「人間の承認ゲート」として規定したら、
  エージェント側の「どこまで自走してよいか」の判断が安定した(承認前にタグを打たない等)
  (記録: 2026-07-20)。
- 知見: 自動化の境界は能力ではなく権限で引く。「判断は委譲不可、作業は委譲可」の言い方が
  3モデルとも誤読なく通った。
- 反映: release.md / skill-install / 編成表に反映済み。

## 運用の記録先を変えるときは旧参照を全文棚卸しする

- 事実: 記録先を verification.md から PR 本文へ変えた際、旧参照が 7 doc・11箇所に
  残っており、クロスレビューが一括検出した(記録: 2026-07-20)。
- 知見: 「どこに記録するか」は多数の doc が暗黙に依存する横断規定。変更時は
  旧記録先の名前で全文 grep して棚卸しするのを必須手順にする。
- 反映: 未反映(本ファイルが正)。

## 委譲のバッチサイズは人間が監督できる粒度を優先する

- 事実: 並列の大量委譲(agy 6連発)はユーザーに却下された。1件ずつ順番の実行で以後問題なし
  (記録: 2026-07-08)。
- 知見: 技術上の最適(並列)より、人間が中断・軌道修正できる粒度を優先する。
- 反映: 未反映(運用慣行としてここが正)。
