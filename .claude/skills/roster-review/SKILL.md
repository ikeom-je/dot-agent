# roster-review — 編成の合議と改訂案づくり

「どのモデルがどの領域に強いか」を、3モデルの自己評価+実績で突き合わせ、
**人間が判断できる改訂案**に落とす手順。指揮者が進行し、
編成表([AGENTS.md](../../../AGENTS.md))の書き換えは**人間の承認後のみ**行う。

## 前提(バイアスへの構え)

- モデルの自己申告は**過大評価バイアス**がある。自己申告は3つの判断材料のひとつに過ぎない:
  1. **自己申告** — 各モデルに領域別の 強/中/弱 と根拠を答えさせる
  2. **実績** — このリポジトリの PR 本文・issue・docs/knowledge/ にある
     実際の成功・失敗記録(委譲失敗、レビュー指摘の質、フォールバック履歴)
  3. **公開情報** — モデル提供元の公表機能(マルチモーダル対応等)。伝聞ベンチは根拠にしない
- 指揮者自身も合議の当事者なので、自分の自己評価にはその旨を明記する。

## 手順

1. **領域リストの準備** — AGENTS.md 編成表から奏者の能力領域を抜き出す。指揮者の**責務**
   (判断・検証・採否)は合議の対象外だが、**指揮者をどのCLIが担うか**は、契約・環境の
   制約があるときに人間への提案として proposal に含めてよい(決めるのは人間)。
   検討したい新領域(例: マルチモーダル)があれば加える。
2. **自己評価の収集(各モデルに同じ質問)** — Codex と Antigravity に
   [cli-routing の「呼び出し方」](../cli-routing/SKILL.md) の手段
   (mcp__codex__codex / antigravity:delegate)で以下を依頼し、Claude 自身も同じ形式で答える:

   ```
   以下の能力領域それぞれについて、あなた(<モデル名>)の得意度を 強/中/弱 で自己評価し、
   1行の根拠を付けてください。確信が無い領域は「不明」と答えてください(過大評価しない)。
   加えて、他の2モデル(<名前>)が明らかに強いと考える領域があれば挙げてください。
   領域: <領域リスト>
   出力: 領域ごとに「<領域>: 強|中|弱|不明 — 根拠」の行のみ。
   ```

   他薦(「他のモデルが強い領域」)を必ず聞く — 自薦より他薦の一致の方が信頼できる。
3. **突き合わせ** — 領域×モデルの評価マトリクスを作り、実績(PR・knowledge の記録)を
   1列追加する。自己申告と実績が食い違う場合は実績を優先する。
4. **合意ラウンド(1回)** — マトリクスと担当案を各モデルに見せ、「異議があれば領域と理由を、
   なければ『合意』を」と1回だけ問う(`max_review_rounds` と同様、往復を増やさない)。
   異議は指揮者が採否を判断し、記録する。
5. **改訂案の提示** — `docs/work/<bolt>/roster-proposal.md` に以下を書き、人間に提示する:
   - 現行→提案の差分表(変更行のみ。変更なしの結論も明記)
   - 領域×モデルの評価マトリクス(自己申告/他薦/実績)
   - 各変更の根拠と、合意状況(全会一致/異議あり+採否)
   - 適用した場合のリスクと、検証方法(次の実案件1件での確認観点)
6. **人間の判断** — 承認されたら [cli-routing の編成変更手順](../cli-routing/SKILL.md)
   (編成表の書き換え+ADR+実案件1件で検証)を実施する。却下・保留は proposal に記録して
   bolt を閉じる。**承認前に AGENTS.md を書き換えない。**

## 実施の目安

- モデルのメジャー更新時、委譲失敗・品質低下が続いたとき、新領域(画像・スライド等)を
  編成に加えたいとき。定期実施は不要(合議自体にもコストがかかる)。
- 委譲先が応答しない場合は [cli-routing](../cli-routing/SKILL.md) のフォールバック規則に
  従い、その旨を proposal の判断材料欄に記録する(応答できないこと自体が実績情報)。
