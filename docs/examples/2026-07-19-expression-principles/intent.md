# intent: expression-principles

プロファイル: standard

- 解きたい課題: 成果物の「何をどこに書くか」の原則が無く、コメントに How の説明、
  コミットログに What の列挙、実装の写しのようなテストが混入しうる。
- この bolt でやること: 表現の4原則(コード=How / テストコード=What /
  コミットログ=Why / コードコメント=Why not)をプロセス全体に織り込む:
  1. AGENTS.md の常時前提原則に追加(全エージェントに効く単一の定義箇所)
  2. git.md のコミット規約に「本文は Why」を具体化
  3. test.md に「テストは What を語る」を具体化
  4. review.md のコード観点に原則違反の検出を追加
  ※ 内部改善boltのため上流フェーズ(2〜5)は省略。
- やらないこと: 既存コード(tools/)の遡及修正、lint 等による機械的強制。
- 完了の定義: 4原則が AGENTS.md に定義され、git/test/review の各docが自分の担当分を
  具体化して参照している。正式グリーン判定グリーン、相互レビュー収束、merge 済み。
