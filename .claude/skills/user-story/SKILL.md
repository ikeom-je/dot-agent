# user-story — ユーザーストーリー作成

「誰の・どんな行動を・なぜ支援するか」を、調査の裏付けとともに言語化する。
テンプレート: [docs/product/templates/user-story.md](../../../docs/product/templates/user-story.md) を
bolt の work ディレクトリに `user-story.md` としてコピーして書く。

## 手順

1. **ペルソナを調査から作る** — ux-research.md の発見(支持された仮説)からペルソナを組み立てる。
   **調査していない属性を創作しない**。裏付けのない項目は「(未調査)」と明記する。
   UX調査が無い bolt では、ペルソナ全体を仮説として扱い、その旨を明記する。
2. **ストーリーマップの背骨を引く** — ユーザーの活動を時系列に左から右へ並べる。
   機能一覧ではなく、ユーザーの行動の流れで書く。
3. **スライスを切る** — MVP/次/後 の3段で優先度を付ける。MVPは「背骨を最初から最後まで
   一本通る最小のストーリー群」にする(特定の活動だけ厚くしない)。
4. **MVPスライスのみ詳細化** — 受け入れ条件を検証可能な形で書く(YAGNI: 次/後は詳細化しない)。
   各ストーリーに根拠となる調査結果(発見/仮説番号)を紐付ける。

## 検証(セルフチェック)

- [ ] 各ストーリーは「誰が・なぜ・何を」の形か(機能名の言い換えになっていないか)
- [ ] ペルソナの各属性に調査の裏付け(または「未調査」の明記)があるか
- [ ] MVPスライスは背骨を一本通っているか
- [ ] 受け入れ条件は第三者が判定できる形か

## 相互レビュー(収束ゲート)

ミニループの相互レビュー(`review_gate_phases` の対象フェーズの場合。[review.md](../../../docs/process/review.md)。観点: 調査との整合・
ストーリーの粒度・受け入れ条件の検証可能性)を収束させてフェーズ完了。

## 完了後の導線

- 画面・体験を伴うプロダクト → `ux-design` スキルへ
- API・CLI など UI の無いプロダクト → UX設計を省略し spec.md(要件)へ。`bolt` スキルに戻る
