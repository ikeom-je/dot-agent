# workflows.md — ユースケース別ワークフロープロファイル

同じ bolt 構造でも、案件の性質によって「どのフェーズを厚くするか・収束パラメータを
どう振るか」は変わる。ここでは代表的なプロファイルを定義する。

## 使い方

1. bolt 開始時(または AGENTS.md の「プロジェクト固有」記入時)にプロファイルを1つ選び、
   **standard を含めて** intent.md に「プロファイル: <名前>」と1行書く(選んだ/選んで
   いないを区別可能にする)。
2. プロジェクト全体で使う場合は AGENTS.md の収束パラメータ表の値を推奨値に書き換える。
   bolt 単位の一時調整は、AGENTS.md の収束パラメータ節の例外規定に従い intent.md の
   明記をもって適用する。
3. 迷ったら `standard`(既定値のまま)でよい。プロファイルは出発点であり、
   途中で性質が変わったら次の bolt から切り替える。

## プロファイル一覧

### standard(既定)

- 使う場面: 通常の機能開発・企画。
- フェーズ構成: lifecycle.md の全フェーズ(省略ルールは lifecycle.md 通り)。
- パラメータ: AGENTS.md の既定値のまま。

### design-first(計画と設計が重要な案件)

- 使う場面: 公開API・課金・データモデル変更など、後戻りコストが大きい案件。
- フェーズ構成: lifecycle.md 通り(その省略ルールは適用してよい。例: API 案件の UX設計省略)。
  ただし設計フェーズでは design-doc(2〜3案比較)+ ADR を必須にし、
  spec と plan のレビューを最重視する。
- パラメータ推奨値: `max_review_rounds`=3 / `review_gate_phases`=all / `max_phase_bounce`=3
- 注意: 実装の速さより「spec・設計docの曖昧さゼロ」を優先。レビューで設計案の比較根拠を
  必ず問う。

### mvp-sprint(MVPを早く出したい案件)

- 使う場面: 価値仮説をとにかく早く市場で確かめたい。作り直し前提。
- フェーズ構成: 上流は intent+最小 spec(スコープ/非スコープ/成功基準のみ)に圧縮。
  PRFAQ〜UX設計は省略可(省略を intent に1行残す)。ユーザーストーリーだけは
  MVPスライス1行群として spec 内に書く。
- パラメータ推奨値: `max_review_rounds`=1 / `review_gate_phases`=コードのみ / `max_fix_loops`=2
- 注意: 省略した上流は「やらない」のではなく「負債として記録する」。MVP検証後の
  本実装 bolt で design-first に切り替えて返済する。テストを弱める禁止事項は緩めない。

### pitch-deck(アイデアを共有するスライドを作りたい案件)

- 使う場面: 社内提案・壁打ち・投資家/上長へのアイデア共有。実装しない。
- フェーズ構成: 企画(intent)→ PRFAQ(または最小の課題/解決メモ)→ スライド構成
  ([pitch-deck テンプレート](../product/templates/pitch-deck.md))で収束する上流bolt。
  データが必要なら ux-research を薄く(デスクリサーチのみ)挟む。
- パラメータ推奨値: `max_review_rounds`=1 / `review_gate_phases`=all(ドキュメント観点)
- 注意: スライドの主張には出典または「仮説」の明記を必須にする(cross-review の
  調査資料観点でレビューする)。ビジュアルデザインはこのフレームワークの対象外。

## プロファイルの追加

新しいユースケースが繰り返し発生したら、この表に「使う場面/フェーズ構成/パラメータ
推奨値/注意」の4点でプロファイルを追記する(それ自体を bolt として回す)。
