# prfaq — PRFAQ 作成

Working Backwards: 顧客に届く瞬間から逆算して企画を言語化する。
テンプレート: [docs/product/templates/prfaq.md](../../../docs/product/templates/prfaq.md) を
bolt の work ディレクトリに `prfaq.md` としてコピーして書く。

## 書く順序

1. **顧客課題** — intent.md の課題を、具体的な顧客のシーンとして書き直す。
   ここが弱いと以降すべてが空中戦になる。競合・現状の代替手段の確認が必要なら
   `ux-research` スキル(デスクリサーチ)を先に薄く回す。
2. **プレスリリース** — 見出し→解決→顧客の声→始め方。顧客の言葉だけで書く。
   技術用語・社内用語が混ざったら書き直し。
3. **外部FAQ** — 顧客が購入/利用前に必ず聞くこと(価格・既存との違い・データの扱い)を最低3つ。
4. **内部FAQ** — 実現性への懸念(なぜ今か・最大のリスク・成功指標・技術的難所)に正面から答える。
   **答えられない質問こそ書き残す**。それが次の UX 調査・検証の仮説になる。

## 検証(セルフチェック)

- [ ] 見出しは顧客価値を言っているか(機能名の紹介になっていないか)
- [ ] 課題は特定の顧客の具体的なシーンか(「みんな困っている」は書き直し)
- [ ] 成功指標は測定可能か
- [ ] 内部FAQに「答えられていない質問」が正直に残っているか

## 相互レビュー(収束ゲート)

`review_gate_phases` がこのフェーズを対象とする場合、セルフチェック後に
[docs/process/review.md](../../../docs/process/review.md) に従い別モデル
(cross-reviewer subagent または antigravity:review)にレビューさせる。
往復は `max_review_rounds` まで。レビュー収束をもってフェーズ完了。

## 完了後の導線

- 検証が必要な仮説が残っている → `ux-research` スキルへ
- 仮説が十分固い(または検証済み) → `user-story` スキル(ユーザーストーリーフェーズ)へ
- PRFAQ 自体の見直しが大きい → この bolt を閉じて、修正版を新しい bolt で
