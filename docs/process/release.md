# release.md — リリースの判定と手順

リリースは **bolt 横断の節目**であり、bolt 内のフェーズではない。複数の bolt が main に
merge された状態を、外(利用者・顧客・チーム)に出せる形に束ねて印を付ける行為。
プロジェクト種別ごとの固有手順(デプロイコマンド、ストア申請等)はここには書かず、
[AGENTS.md](../../AGENTS.md) の「プロジェクト固有」欄に記入する。

## リリース判定チェックリスト(GO の前提)

- [ ] 対象範囲の bolt がすべて収束している(verification.md、上流だけの bolt は
      intent.md 末尾の引き継ぎメモがあり、未解決指摘・エスカレーション中の項目が無い)
- [ ] 正式グリーン判定(AGENTS.md プロジェクト固有欄のテストコマンド)を
      オーケストレーターがクリーンな main 上で実行してグリーン
- [ ] リリースノートが書けている(下記)
- [ ] ロールバック手順が書けている(下記)
- [ ] **GO 判断は `escalation_to`(人間)が行う**。エージェントの仕事はチェックリストを
      揃えて提示するまで。**GO 前にタグ付け・デプロイ・公開を行わない**。
      GO 後の実行作業(タグ付け・デプロイ)は、人間の指示があればエージェントが行ってよい
      (判断は委譲不可、作業は委譲可)

## バージョニングとタグ

- 既定は Semantic Versioning(`vMAJOR.MINOR.PATCH`)。バージョン番号の解釈が
  なじまないプロジェクト(ドキュメント・社内ツール等)は日付タグ(`release-YYYY-MM-DD`)
  でよい。どちらを使うかは AGENTS.md プロジェクト固有欄に1行書く。
- タグは人間の GO 後に、人間(または指示を受けたエージェント)が main の merge コミットに
  付ける(`git tag -a <tag> -m "<要約>"`)。

## リリースノートの作り方

- 前回リリースタグ以降のコミットログから作る: `git log <前回タグ>..HEAD --oneline`
  (前回タグが無い初回リリースは全履歴 `git log --oneline` を対象とする)
- コミットログには Why が書かれている(表現の原則・[git.md](git.md))ので、
  **Why を利用者の言葉に翻訳して束ねる**。「何を変えたか」の列挙ではなく
  「利用者にとって何が良くなるか・何に注意が必要か」で書く。
- 構成: ハイライト(1〜3個)/ 変更点 / 非互換・移行が必要な点 / 既知の問題。
- 対象読者(エンドユーザー/開発者/社内)に合わせた文体はプロジェクト固有欄の記載に従う。

## ロールバック方針

- リリース前に「戻し方」を1〜3行で書けること自体を GO の条件にする(上記チェックリスト)。
  書けないリリースは設計に問題がある。
- 既定の戻し方: 直前タグへの再デプロイ(サービス)/ 前バージョンの再公開(配布物)/
  revert コミット(ドキュメント・ライブラリ)。データマイグレーションを含む場合は、
  戻せない変更(不可逆)を リリースノートの非互換欄に明示する。

## リリース bolt

リリース作業自体も小さな bolt として回してよい(`bolt/YYYY-MM-DD-release-<tag>`)。
その場合の成果物は release-notes と検証記録(チェックリストの結果)であり、
上流フェーズは省略(intent に1行)。リリース後の不具合対応は新しい bolt として起票する。
