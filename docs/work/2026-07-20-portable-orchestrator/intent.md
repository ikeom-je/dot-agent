# intent: portable-orchestrator

プロファイル: standard

- 解きたい課題: 指揮者(オーケストレーター)が Claude Code 固定で、組織・個人の契約状況に
  よっては dot-agent 自体が使えなくなる(契約リスク)。また SKILL・subagent 定義の
  呼び出しが Claude Code のハーネス(Skillツール・MCP)前提で書かれており、
  他CLIが指揮者のとき使えない。
- この bolt でやること:
  1. AGENTS.md: 指揮者行を「既定: Claude Code、契約・環境の制約時は交代可(編成変更手順+
     実案件検証)」に緩和。指揮者/奏者の責務をロールベースで集約した「役割の特定」節を追加
     (どのCLIも編成表で自分の役割を特定して振る舞う)
  2. SKILL・subagent 定義の汎用利用を明文化: いずれも全CLI共通のMarkdown定義であり、
     自動発見が無いCLIは「ファイルを読んで従う/本文を指示として渡す」で使う
  3. cli-routing の呼び出し方を2層化: Claude Code 指揮時の最適経路(MCP・プラグイン)と、
     任意の指揮者で使える汎用経路(ヘッドレスCLI+ファイルベース委譲)のマトリクス
  4. プロセスdoc・スキルの「(Claude)」直書きをロール名(指揮者)に正規化
  5. CLAUDE.md / GEMINI.md を対称化(役割は編成表で特定、既定役割のみ記載)
  ※ 内部改善boltのため上流フェーズ(2〜5)は省略。
- やらないこと: 指揮者の自動認識・自動切替、Codex/agy 用のハーネス実装(skills自動発見等)、
  各CLIのヘッドレスコマンドの動作保証(交代時の実案件検証で確認する)。
- 完了の定義: 「(Claude)」のロール直書きがゼロ(既定編成の例示を除く)、任意指揮者向けの
  汎用呼び出し経路が定義済み、正式グリーン判定グリーン、相互レビュー収束、merge 済み。
