# test.md — テストループの回し方と収束条件

開発とテストの往復を「最小の回数で収束させる」ためのルール。
無限の修正ループと、テストを弱めて通すごまかしの両方を防ぐ。

## テストコードの原則(What を語る)

- **テストコードには What を書く**(AGENTS.md 表現の原則): テスト名は「何を満たすべきか」が
  仕様として読める文にし、アサーションは振る舞い(入力→期待される出力・状態)を検証する。
- 実装の内部手順をなぞるテスト(実装の写し)は書かない。実装をリファクタしたら壊れる
  テストは、What ではなく How を検証してしまっている兆候。

## テストループの手順

1. **実装** — plan.md のタスク単位で実装する(可能なら失敗するテストを先に書く)。
   コードは How をコード自身で語らせ、コメントは Why not(別のやり方をしなかった理由・
   自明でない制約)に限る。
2. **実行** — テストを実行し、結果をそのまま記録する。
3. **失敗分析** — 失敗したら、修正の前に「なぜ失敗したか」を1〜2行で言語化する。
   仮説なしに手を動かさない。
4. **修正** — 分析に基づく最小の修正を行い、2 に戻る。

## 収束条件(停止ルール)

- **修正ループは `max_fix_loops` 回まで**([AGENTS.md](../../AGENTS.md) の収束パラメータ)。
  同じテスト(または同じ原因系)への修正が上限回でも失敗したら、ループを止める。もう1回は試さない。
- 停止したら、`verification.md` に以下を書いて **`escalation_to` にエスカレーション**する:
  - 失敗しているテストと症状
  - 試した修正と、それぞれの仮説・結果
  - 現時点で最も疑わしい原因
- 上限は「同じ問題への挑戦回数」。別の独立したテスト失敗はカウントを分ける。

## グリーン判定の権限

- **グリーンの最終判定は指揮者(オーケストレーター)が行う**。subagent(奏者)の
  「テストは通りました」という自己申告を判定に使わない。
- 判定時はクリーンな状態で実行する: 未コミットの一時変更・環境の書き換えが無いことを
  `git status` で確認してからテストコマンドを実行する。
- subagent が環境側を書き換えて(依存パッケージへのパッチ、モックによる置き換え等)テストを
  通している兆候があれば、diff 全体を確認する。

## 禁止事項(テストを弱める修正)

失敗を消すために以下をしてはならない:

- テストの skip / disable / コメントアウト
- アサーションの削除・緩和(期待値をテスト対象の実出力に合わせて書き換える、を含む)
- 本来の依存を無断でモックに置き換えて通す
- タイムアウトやリトライの追加で不安定さを隠す

テスト自体が間違っていると判断した場合は、「テストが間違っている根拠」を verification.md に
書いた上で修正する。根拠が書けないなら、それは実装側の問題である。
