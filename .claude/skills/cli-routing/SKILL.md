# cli-routing — エージェントCLIの使い分け

指揮者(既定: Claude Code。編成表で特定)がルーティングを判断し、それ以外が奏者。
委譲は目的ではなくコスト・品質の最適化手段。**迷ったら指揮者が自分でやる。**

## 役割の責務(全CLI共通)

- **指揮者**: bolt の進行(bolt スキル)/ 委譲判断(このスキル)/ グリーン判定を自ら実行 /
  レビューの採否判断 / `escalation_to` への報告。
- **奏者**: 渡された書面(spec / plan / エージェント定義)の範囲でのみ作業。結果は
  ダイジェスト(変更ファイル一覧・要約・実行した検証と結果・未解決事項)で返す。
  自己検証は参考情報であり、GREEN の断定報告をしない。

## スキル・エージェント定義の互換性(全CLI共通)

`.claude/skills/*/SKILL.md` と `.claude/agents/*.md` は全CLI共通の Markdown 定義
(置き場所が `.claude/` なだけ)。Claude Code 以外のCLIが使う場合:

- **スキル**: 自動発見されないので、AGENTS.md の参照表からパスを引き、ファイルを読んで従う。
- **エージェント定義**: frontmatter の `tools:`(Claude Code 固有)は無視し、
  本文を役割指示として対象CLIに渡す(手段は下の「呼び出し方」)。

## 担当の決め方

1. タスクの**能力領域**を特定する(実装/バルク/調査/マルチモーダル/レビュー…)。
2. [AGENTS.md のエージェント編成表](../../../AGENTS.md) でその領域の「担当」を引く
   (担当の定義はあの表が唯一。ここには書かない)。
3. 下の損益分岐を超えるときだけ委譲する。超えないなら領域に関わらず自分でやる。

## 委譲の損益分岐

委譲には「仕様の書面化+往復+検証」の固定コストがかかる。以下は**委譲しない**:

- 小さいタスク(1〜2ファイルの編集、リネーム、一行修正)— 往復コストの方が高い
- 判断が重いタスク(要件の解釈、設計のトレードオフ、レビューの採否)— 指揮者の仕事
- 自己完結していないタスク(会話の文脈を大量に引き継がないと説明できないもの)

逆に、仕様が明確で反復的・大量なら、細切れに委譲せず**1回の大きな委譲にまとめる**。

## 委譲時の必須3点

1. **仕様を書面で渡す** — 会話履歴の貼り付けではなく、`spec.md` / `plan.md` へのパスと完了条件を渡す。
   リポジトリを直接読ませる(Antigravity なら `--dir` で AGENTS.md を読ませる)。
2. **ダイジェストで受ける** — 生の出力全文をコンテキストに戻さない。変更ファイル一覧+要約を受け、
   確認は diff に対して行う。
3. **検証は指揮者が行う** — subagent の自己申告 GREEN を信じない。
   [docs/process/test.md](../../../docs/process/test.md) に従い、クリーン状態で自分で実行して判定する。

## 委譲失敗時のフォールバック

委譲先の失敗(空応答・エラー・タイムアウト)への対処:

- 試行は**ひとつの委譲先につき合計 `max_delegation_retries` 回まで**
  (ティア変更・プロンプト言い換えによる再試行も1回と数える)。
- 上限に達したらその委譲先を諦め、**編成表(AGENTS.md)の「代替」列を左から順に**試す。
  代替経路も同じ上限で数える(失敗し続ける委譲先へのリトライ連打をしない)。
- **同一委譲先内でも、モデルが非対応・エラー(例: 「要アップグレード」400エラー)なら
  [AGENTS.md のモデル優先順位](../../../AGENTS.md) の次点を明示指定して再試行する**
  (これも1回と数える。「呼び出し方」のモデル指定例を参照)。
- 最後の代替まで失敗したら、bolt の該当タスクを中断して `escalation_to` に報告する。

フォールバックしたこと・理由は PR 本文に1行残す。

## 編成の変更手順(モデルの強み・得意領域が変わったとき)

適用対象は**運用開始後の担当差し替え・領域追加**(テンプレート導入時の初期記入は対象外。
初期記入は README の導入手順に従い編成表を直接編集してよい)。

1. AGENTS.md のエージェント編成表の「担当」「代替」列を書き換える(領域の行が無ければ追加)。
2. 下の「呼び出し方」対応に、新担当への呼び出し手段があるか確認する。無ければ
   subagent 定義(`.claude/agents/`)や利用プラグインを追加・調整する。
3. 変更理由(どのモデルのどの強みを根拠にしたか)を ADR
   ([docs/design/templates/adr.md](../../../docs/design/templates/adr.md))として1枚残す。
4. 変更自体を bolt として回し、次の実案件1件で新担当の品質を検証してから恒久化する。

## 呼び出し方(エージェント→手段の対応)

委譲は常に**ファイルベース**(spec/plan/エージェント定義のパス+完了条件を渡し、
ダイジェストで受ける)。これはどの指揮者でも同じ。手段だけが環境で変わる:

### 指揮者が Claude Code のとき(最適経路)

呼び出しには [AGENTS.md のモデル優先順位](../../../AGENTS.md) の第一候補モデルを明示指定する
(未指定だと環境のデフォルトモデルが動き、非対応エラーになることがある)。

- **Codex へ**: `codex-worker` subagent([.claude/agents/codex-worker.md](../../agents/codex-worker.md))、
  またはレビューなら mcp__codex__codex(read-only)。`model` パラメータに
  モデル優先順位の第一候補を指定し、非対応エラーなら次点を指定して再試行する。
- **Antigravity へ**: `antigravity:delegate` スキル(または antigravity-delegate subagent)。
  Web調査は `antigravity:research`、クロスレビューは `antigravity:review`。
  `agy-delegate` はモデル優先順位の第一候補、失敗時は次点を指定する
  (ティア指定 `--tier` とは別軸。速度優先のバルク処理は `--tier flash` と
  次点モデルを併用してよい)。
  マルチモーダル(画像生成・検出/比較・スライド生成)も同経路で委譲し、
  成果物(画像・スライド)は指揮者が実物を開いて検証する。
- **クロスレビュー**: `cross-reviewer` subagent。運用ルールは
  [docs/process/review.md](../../../docs/process/review.md)。

### 任意の指揮者で使える汎用経路(ヘッドレスCLI)

MCP・プラグイン・subagent 機構が無い環境では、対象CLIのヘッドレス実行に
「エージェント定義の本文+書面のパス+完了条件」を指示として渡す:

| 呼び出し先 | コマンド例(バージョンにより異なる。交代時の実案件検証で要確認) |
|---|---|
| Claude | `claude -p "<指示>"` |
| Codex | `codex exec "<指示>"`(read-only レビューはサンドボックスオプションを併用) |
| Antigravity | `agy -p "<指示>"`(または環境のラッパー。Web検索系は yolo/承認オプションが必要な場合あり) |

- 指示の型: 「まず AGENTS.md を読み、次に <エージェント定義のパス> の本文を役割として、
  <spec/plan のパス> のタスクを実行し、ダイジェスト(変更ファイル一覧・要約・実行した
  検証と結果・未解決事項)だけを出力せよ」。
- ヘッドレス実行は1呼び出し=1往復。継続スレッドが無いため、**再依頼(レビューの2往復目
  など)は新規呼び出しに前回の指摘・修正内容を引用して行う**。cross-review の
  「同一スレッドで再依頼」はスレッド機能がある経路(MCP等)のみの規定であり、
  往復回数の上限(`max_review_rounds` 等)は経路に関わらず同一に適用する。
- 委譲・検証・採否の**規律(必須3点・フォールバック・グリーン判定)は経路に関わらず同一**。
