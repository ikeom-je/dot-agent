# cross-review — クロスモデルレビューの実行手順

[docs/process/review.md](../../../docs/process/review.md) の運用ルールを実行に落とす手順。
レビュアーの選択(別モデル原則)・観点・収束条件は review.md に従う。

## プロトコル(往復は `max_review_rounds` まで)

1. **依頼(1往復目)** — レビュアーを read-only で起動し、下のテンプレートで依頼する。
   - Codex: mcp__codex__codex(`sandbox: read-only`, `approval-policy: never`, `cwd` 指定)
   - Antigravity: `antigravity:review` スキル
   - どちらも不能: cross-reviewer subagent(同モデルになるため、その旨を記録に残す)
2. **採否判断** — 指摘ごとに採用/部分採用/棄却を決める。
   **失敗シナリオ付きの指摘は、反映する前に必ず再現を試みる**(コードなら再現テスト、
   ドキュメントなら該当箇所の読み直し)。再現しない指摘は前提の誤りとして棄却してよいが、
   同じ箇所に別の正当な問題(spec不整合等)が含まれていないかを確認してから捨てる。
3. **修正** — 採用した指摘を反映する。コードは可能ならテスト先行(RED→GREEN)で
   ([docs/process/test.md](../../../docs/process/test.md) に従う)。
4. **再依頼(2往復目以降)** — 同一スレッド(codex-reply 等)で「何をどう直したか+棄却した
   指摘とその根拠」を伝えて再レビューさせる。`max_review_rounds` 往復目には
   「最終往復。残存する major 以上のみ、無ければ『収束』と回答」と明示する。
5. **記録** — PR 本文(ローカル運用では merge コミット本文)に、指摘件数・採否と根拠・
   往復数を書く。`max_review_rounds` で打ち切った未解決指摘は `escalation_to` へ。

## レビュー依頼テンプレート

共通の前書き(そのまま使う):

```
あなたはレビュアーです(読み取り専用、修正禁止)。docs/process/review.md の観点で
<対象> をレビューしてください。
対象: <ファイルパス or「git diff main で取得」>
照合先: <intent.md / spec.md / 前フェーズ成果物のパス>
出力形式: `- [critical|major|minor] <ファイル:行 or セクション> 指摘 — 失敗シナリオ/根拠`
のリスト。指摘ゼロならその旨。スタイルの好みは挙げない。新機能の提案は指摘に混ぜない。
```

観点はレビュー対象に応じて review.md から具体化して足す:

- **コード**: 正しさ(エッジケース: 空入力・境界・パスのスペース等。具体的な失敗シナリオを
  添えられる指摘のみ)/ spec との整合 / テストの健全性(期待値合わせ・ケース欠落)/ YAGNI /
  表現の原則違反(定義は AGENTS.md 原則4)
- **ドキュメント**: 前フェーズとの整合 / 曖昧で2通りに読める規定 / 根拠のない断定 /
  答えたふりの記述 / YAGNI
- **調査資料**: 発見と解釈の分離 / 判定が発見から論理的に導けるか / 出典の有無 /
  設問の誘導性

## 実証済みの注意点

- **レビュアーの検証環境は信用しない** — レビュアー(Codex等)はサンドボックス制約で
  テストを実行できないことがある。グリーン判定は必ずオーケストレーター自身が行う
  ([docs/process/test.md](../../../docs/process/test.md))。
- **指摘の失敗シナリオは間違っていることがある** — 前提が誤った major 指摘でも、
  周辺の spec 不整合の指摘は正当、というケースが実際にあった。機械的に全採用も全棄却もしない。
- **再依頼は同一スレッドで** — 新規スレッドだと文脈を失い、初回と同じ指摘を繰り返す。
  スレッド機能が無い経路(ヘッドレスCLI)では、新規呼び出しに前回の指摘と修正内容を
  引用して代替する(cli-routing の汎用経路)。
- **レビュー依頼前に新規ファイルを `git add` する** — untracked ファイルは `git diff` に
  現れず、レビュー対象から欠落する。
- 複数の密結合な成果物(user-story と ux-design 等)は1回の依頼にまとめてよい
  (指摘・採否はファイル単位で記録する)。
