# docs/knowledge/ — 知見の永続化(忘却防止)

AI agent のセッション記憶は揮発する(コンテキスト圧縮・セッション終了・モデル切替で、
会話の中にしか無い判断や学びは消える)。**ツール依存の memory 機能に頼らず、
このディレクトリのファイルを知見の正**とする。ファイルにある知見だけが次のセッション・
別のエージェントに引き継がれる。

## dump のタイミング(機構)

1. **bolt 収束時(必須チェック)** — merge 前に「この bolt で得た、リポジトリの成果物に
   残っていない知見はあるか?」を自問し、あれば [insights.md](insights.md) に追記、
   無ければ PR 本文に「knowledge dump: なし」と1行書く
   (git.md マージ収束ルールの手順4)。
2. **セッションの区切り** — 長いセッションを畳む前・コンテキスト圧縮が近いと感じたとき、
   未記録の知見を dump してから畳む。
3. **失敗・想定外の直後** — 委譲失敗、レビューでの重要な見落とし、ツールの落とし穴は
   記憶が新しいうちに書く(後でまとめて、は忘却する)。

## 何を書くか / 書かないか

- **書く**: 成果物から読み取れない知見 — 失敗と回避策、ツール・モデルの癖の実測、
  やってみて分かった判断基準、棄却した選択肢とその理由。
- **書かない**: リポジトリの成果物・git 履歴・プロセス doc から分かること(重複させない)。
  知見が規定に昇格したら(プロセス doc に反映したら)、insights 側には「→ <doc> に反映済み」と
  印を付ける。

## 形式

[insights.md](insights.md) に追記式で書く。1知見 = 1見出し(**見出しはテーマで付ける。
日付ベースの整理はしない** — 記録日は本文のメタ情報):

```markdown
## <テーマの一言タイトル>
- 事実:(何が起きたか。1〜3行。記録日を添える)
- 知見:(何を学んだか・次からどうするか)
- 反映:(プロセス doc に昇格済みなら参照。未反映なら「未反映」)
```

肥大化したらテーマ別ファイル(`insights-<topic>.md`)に分割してよい(このREADMEから
リンクする)。古くなった知見は消さず「(失効: 理由)」を付ける。

## 知見の3層(探索モデル)

知見は insights.md だけでなく3つの層に分散している。何かを思い出したい/調べたいときは、
`tools/search-knowledge.sh <キーワード>` で3層を横断検索する(追加依存なし。SQLite等の
DBは、この3層検索でも規模的に不十分になったときの次の一手として温存する)。

| 層 | 内容 | 単体で調べる手段 |
|---|---|---|
| **insights.md** | 人間が選別した恒久知見(成果物から読み取れないもの) | 直接読む / grep |
| **git log の Why** | 全コミットログ(表現の原則で Why が書かれている) | `git log --all --grep=<語> -i` |
| **issue / PR** | 議論の経緯・却下した提案・エスカレーション記録 | `gh issue list --search <語> --state all` / `gh pr list` |

**Decision Shadow(却下した代替案の消失)に注意する**: git commit は diff を残すが
「なぜ他の案にしなかったか」を残さないことがある。重要な設計判断の却下・変更は
[docs/design/templates/adr.md](../design/templates/adr.md) に却下した代替案として残す
([cross-review スキル](../../.claude/skills/cross-review/SKILL.md) 参照)。
