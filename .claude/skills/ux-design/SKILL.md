# ux-design — UX設計

ストーリーを「どんな体験・画面の流れで実現するか」に落とす。**見た目のデザインはしない**
(構造・フロー・情報の優先順位まで)。
テンプレート: [docs/product/templates/ux-design.md](../../../docs/product/templates/ux-design.md) を
bolt の work ディレクトリに `ux-design.md` としてコピーして書く。

## 手順

1. **体験の方針を1〜3行で決める** — この体験で譲れない設計判断を先に言語化する。
   以降の画面設計はこの方針との整合で判断する。
2. **画面フローを引く** — MVPスライスのストーリーごとに、入口→目的達成の遷移を描く。
   ハッピーパスだけでなく、主要な離脱・エラー分岐も含める。
3. **情報設計** — 各画面は1画面1目的。優先表示の順序と主アクション(1つ)を決める。
   迷ったらペルソナのゴール(user-story.md)に立ち返る。
4. **ワイヤーは必要最小限** — 判断が割れる画面のみテキストワイヤーで構造を示す。
   全画面を描かない(YAGNI)。
5. **状態を網羅する** — 空状態・エラー・権限なしで何を見せるかを表で埋める。
   ここの欠落が実装フェーズの手戻りになる。
6. **未解決の設計課題を明記** — プロトタイプ検証が必要な仮説は ux-research の追加仮説として
   起票し、要件・実装に引き継ぐ判断は明示的に書く。

## 検証(セルフチェック)

- [ ] 全MVPストーリーに対応する画面フローがあるか(逆に、ストーリーに無い画面を作っていないか)
- [ ] 各画面の目的は1つか、主アクションは1つか
- [ ] エラー・空状態が埋まっているか
- [ ] 体験の方針と矛盾する画面が無いか

## 相互レビュー(収束ゲート)

`review_gate_phases` がこのフェーズを対象とする場合、
[docs/process/review.md](../../../docs/process/review.md) に従い別モデルにレビューさせる
(観点: ストーリーとの過不足・フローの破綻・状態の網羅)。
往復は `max_review_rounds` まで。レビュー収束をもってフェーズ完了。

## 完了後の導線

- spec.md(要件)へ。`bolt` スキルに戻る。UX設計の「未解決の設計課題」を spec の
  スコープ判断に引き継ぐ
